## 🎯 Классификация инцидентов

### Уровни серьёзности

```
┌─────────────────────────────────────────────────────────────────┐
│                         INCIDENT LEVELS                         │
├─────────┬───────────────────────────────────────────────────────┤
│   P0    │  CRITICAL — Полная недоступность, потеря данных       │
├─────────┼───────────────────────────────────────────────────────┤
│   P1    │  HIGH — Критичные функции не работают                 │
├─────────┼───────────────────────────────────────────────────────┤
│   P2    │  MEDIUM — Частичная недоступность, замедление         │
├─────────┼───────────────────────────────────────────────────────┤
│   P3    │  LOW — Косметические проблемы, отдельные баги         │
└─────────┴───────────────────────────────────────────────────────┘
```

### Критерии классификации

| Критерий                   | P0           | P1           | P2          | P3         |
| -------------------------- | ------------ | ------------ | ----------- | ---------- |
| **Пользователи затронуты** | 100%         | > 50%        | 10-50%      | < 10%      |
| **Потеря данных**          | Да           | Возможна     | Нет         | Нет        |
| **Выручка**                | Критична     | Значительна  | Умеренна    | Минимальна |
| **Репутация**              | Высокий риск | Средний риск | Низкий риск | Нет риска  |
| **Время реакции**          | 15 мин       | 30 мин       | 2 часа      | 24 часа    |

### Примеры инцидентов

**P0 — Critical:**

- 📉 Сайт полностью недоступен
- 🔐 Утечка данных пользователей
- 💾 Потеря данных заказов
- 💳 Все платежи не работают
- 🗄️ База данных недоступна

**P1 — High:**

- 🛒 Оформление заказов не работает
- 🔐 Аутентификация не работает
- 📦 Конфигуратор ПК не работает
- 💳 Часть платежей не проходит
- 📧 Email уведомления не отправляются

**P2 — Medium:**

- 🐌 Медленная загрузка страниц
- 🔍 Поиск не работает
- 📱 Мобильная версия сломана
- 📊 Отчёты в админке не работают
- 🎨 Проблемы с отображением

**P3 — Low:**

- 📝 Опечатки на сайте
- 🎨 Косметические проблемы UI
- 📎 Не работают второстепенные функции
- 📄 Проблемы с документацией

## 👥 Роли и обязанности

### On-Call Engineer (Дежурный)

**Обязанности:**

- ✅ Первый ответ на алерт
- ✅ Начальная классификация
- ✅ Эскалация при необходимости
- ✅ Координация команды
- ✅ Коммуникация статусов

**Доступы:** L3

**Смена:** 1 неделя, ротация

### Incident Commander (Командир инцидента)

**Обязанности:**

- ✅ Управление процессом решения
- ✅ Распределение задач
- ✅ Принятие решений
- ✅ Коммуникация со стейкхолдерами

**Кто:** Tech Lead или Senior разработчик

### Communications Lead (Коммуникации)

**Обязанности:**

- ✅ Обновление статус страницы
- ✅ Коммуникация с клиентами
- ✅ Внутренние апдейты
- ✅ Post-Mortem публикация

**Кто:** PM или назначенный разработчик

### Scribe (Секретарь)

**Обязанности:**

- ✅ Ведение хронологии
- ✅ Документирование решений
- ✅ Сбор логов и скриншотов
- ✅ Помощь в Post-Mortem

**Кто:** Младший разработчик (для обучения)

## 📚 Playbooks

### P0: Сайт полностью недоступен

**Симптомы:**

- Uptime алерт
- 502/503/504 ошибки
- Пользователи сообщают о недоступности

**Действия:**

```bash
# 1. Проверка статуса
curl -I https://1000fps.ru
curl -I https://api.1000fps.ru

# 2. Проверка сервисов
sudo systemctl status nginx
sudo systemctl status 1000fps-storefront
sudo systemctl status 1000fps-api
sudo systemctl status postgresql
sudo systemctl status redis

# 3. Проверка логов
sudo tail -100 /var/log/nginx/error.log
sudo journalctl -u 1000fps-storefront -n 100 --no-pager
sudo journalctl -u 1000fps-api -n 100 --no-pager

# 4. Быстрое восстановление
sudo systemctl restart nginx
sudo systemctl restart 1000fps-storefront
sudo systemctl restart 1000fps-api

# 5. Проверка БД
sudo -u postgres psql -d 1000fps_prod -c "SELECT 1;"

# 6. Проверка Redis
redis-cli ping

# 7. Если не помогло — проверка ресурсов
df -h
free -m
docker ps
docker logs 1000fps-prod-api --tail 100

# 8. Rollback (если недавний деплой)
cd /var/www/1000fps
git log --oneline -5
git revert HEAD
sudo systemctl restart 1000fps-storefront
sudo systemctl restart 1000fps-api
```

**Коммуникация:**

- T+15min: Статус на status.1000fps.ru — "Investigating"
- T+30min: Апдейт в Telegram канале
- T+60min: Email клиентам (если не решено)

### P1: Оформление заказов не работает

**Симптомы:**

- Ошибки при создании заказа
- Платежи не проходят
- Пользователи сообщают о проблемах

**Действия:**

```bash
# 1. Проверка логов API
sudo journalctl -u 1000fps-api -f | grep -i "order\|payment"

# 2. Проверка платежного шлюза
curl -X GET https://payment-gateway.ru/api/health \
  -H "Authorization: Bearer $PAYMENT_API_KEY"

# 3. Проверка webhook'ов
# Payment dashboard → Webhooks → Recent events

# 4. Тестовый заказ
# Admin → Orders → Create test order

# 5. Проверка БД
sudo -u postgres psql -d 1000fps_prod -c \
  "SELECT status, COUNT(*) FROM orders
   WHERE created_at > NOW() - INTERVAL '1 hour'
   GROUP BY status;"

# 6. Проверка очереди email
redis-cli LLEN mail_queue

# 7. Проверка лимитов
# Payment dashboard → API limits
```

**Временное решение:**

- Включить оплату при получении
- Переключиться на резервный платежный шлюз

---

### P1: Утечка данных

**Симптомы:**

- Сообщение от security исследователей
- Аномальная активность в логах
- Жалобы пользователей

**Действия:**

```bash
# 1. Немедленно отключить затронутый эндпоинт
sudo nginx -s reload  # После добавления deny в конфиг

# 2. Проверка логов доступа
sudo grep "GET /api/v1/users" /var/log/nginx/access.log | tail -100

# 3. Проверка аутентификации
sudo -u postgres psql -d 1000fps_prod -c \
  "SELECT user_id, action, timestamp
   FROM audit_log
   WHERE timestamp > NOW() - INTERVAL '1 hour'
   ORDER BY timestamp DESC
   LIMIT 100;"

# 4. Блокировка подозрительных сессий
sudo -u postgres psql -d 1000fps_prod -c \
  "UPDATE sessions SET expired = true
   WHERE user_id IN (подозрительные_id);"

# 5. Ротация секретов
# - JWT secrets
# - API keys
# - Database passwords
```

**Коммуникация:**

- T+30min: Юридический отдел уведомлён
- T+2h: План коммуникации с пользователями
- T+24h: Email затронутым пользователям

### P2: Медленная загрузка

**Симптомы:**

- Алерт на P95 latency
- Жалобы пользователей
- Низкие показатели Core Web Vitals

**Действия:**

```bash
# 1. Проверка производительности
curl -w "@curl-format.txt" -o /dev/null -s https://1000fps.ru

# 2. Проверка кэша
redis-cli INFO stats | grep hit_rate

# 3. Проверка медленных запросов
sudo -u postgres psql -d 1000fps_prod -c \
  "SELECT query, calls, mean_time
   FROM pg_stat_statements
   ORDER BY mean_time DESC
   LIMIT 10;"

# 4. Проверка CDN
# Cloudflare dashboard → Analytics → Performance

# 5. Проверка размеров страниц
# Chrome DevTools → Network → Waterfall

# 6. Временные меры
# - Включить полный кэш страниц
# - Отключить тяжёлую аналитику
# - Уменьшить качество изображений
```

---

### P2: Конфигуратор ПК не работает

**Симптомы:**

- Ошибки при выборе компонентов
- Проверка совместимости не работает
- Сборки не сохраняются

**Действия:**

```bash
# 1. Проверка логов
sudo journalctl -u 1000fps-api -f | grep -i "config\|compat"

# 2. Проверка кэша совместимости
redis-cli KEYS "compat:*"
redis-cli DEL "compat:cache:*"  # Очистка кэша

# 3. Проверка правил совместимости
sudo -u postgres psql -d 1000fps_prod -c \
  "SELECT COUNT(*) FROM compatibility_rules;"

# 4. Проверка конфигов пользователей
sudo -u postgres psql -d 1000fps_prod -c \
  "SELECT COUNT(*) FROM configs
   WHERE updated_at > NOW() - INTERVAL '1 hour';"

# 5. Временное отключение проверки совместимости
# Feature flag: disable_compat_check = true
```

### Процесс Post-Mortem

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  Schedule   │────▶│   Write     │────▶│   Review    │
│  (T+2h)     │     │  (T+24h)    │     │  (T+48h)    │

```

**Для P0:**

- ✅ Публикация для всей компании

**Для P1:**

- ✅ Обязательно в течение 48 часов
- ✅ Review с Tech Lead
- ✅ Action items задокументированы

**Для P2:**

- 📝 Запись в базу знаний

## 📈 Улучшение процессов

### Метрики инцидентов

| Метрика                         | Цель      | Текущее | Тренд |
| ------------------------------- | --------- | ------- | ----- |
| MTTD (Mean Time To Detect)      | < 5 мин   | 3 мин   | ✅    |
| MTTR (Mean Time To Resolve)     | < 30 мин  | 25 мин  | ✅    |
| MTTA (Mean Time To Acknowledge) | < 10 мин  | 7 мин   | ✅    |
| Инцидентов в месяц              | < 5 P0/P1 | 3       | ✅    |
| Repeat incidents                | 0%        | 5%      | ⚠️    |

### Ежеквартальный review

**Вопросы для обсуждения:**

1. Сколько инцидентов было за квартал?
2. Какие инциденты повторялись?
3. Какие action items не выполнены?
4. Какие процессы нужно улучшить?
5. Какие инструменты нужны?

### Непрерывное улучшение

```
┌─────────────────────────────────────────────────────┐
│              INCIDENT IMPROVEMENT CYCLE             │
├─────────────────────────────────────────────────────┤
│                                                     │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐      │
│   │ Track   │────▶│ Analyze │────▶│  Act    │      │
│   └─────────┘     └─────────┘     └─────────┘      │
│        ▲                                      │     │
│        │                                      │     │
│        └──────────────────────────────────────┘     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**Track:**

- Все инциденты задокументированы
- Метрики собираются автоматически
- Тренды отслеживаются

**Analyze:**

- Ежеквартальный review
- Выявление паттернов
- Приоритизация улучшений

**Act:**

- Внедрение улучшений
- Обновление процессов
- Обучение команды
